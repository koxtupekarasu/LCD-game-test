# DOTMON UI言語 完全仕様

最終更新: 2026-03-15

本ドキュメントは、DOTMONにおけるUI上の記号・語彙・表示原則を統一するための正本メモである。  
各画面ごとの実装都合ではなく、**ゲーム全体で一貫して使うUI言語**を定義する。

---

## 1. 目的

DOTMONのUIは、単なる情報表示ではなく、プレイヤーが短時間で意味を読み取れる**携帯育成液晶ゲーム風の視覚言語**として設計する。

本仕様の目的は以下。

- UI全体の一貫性を保つ
- 記号・略称・表示ルールの乱立を防ぐ
- 予告表示と結果表示を明確に分離する
- 小画面 / 低情報量でも意味が伝わる設計にする
- 今後の FOOD / ITEM / BTTL / TRN / ADV / STAT 拡張でも同じ文法を使えるようにする

---

## 2. UI言語の基本思想

DOTMONのUIは、以下の3層で構成する。

### 2.1 行動の層
プレイヤーが何をするかを示す。

例:
- `STAT`
- `FOOD`
- `TRN`
- `BTTL`
- `ADV`
- `WC`
- `SLEEP`
- `HEAL`
- `EDIT`

### 2.2 状態の層
今どうなっているかを示す。

例:
- HP
- STA
- BREAK
- 充足値
- 安定度
- 体重
- 所持数

### 2.3 意味づけの層
それが何を意味するかを短く補足する。

例:
- ログ
- 説明ペイン
- 効果傾向表示
- 等級表示
- バトル観測ログ

---

## 3. DOTMON UIの基本原則

### 3.1 小さくても読めることを優先する
- 装飾より識別性を優先する
- 細かい情報を増やしすぎない
- 1bit / LCD風表示でも意味が崩れないことを優先する

### 3.2 同じ意味は同じ記号で表す
- 画面ごとに別記号を作らない
- 同じ概念には同じ表示ルールを使う

### 3.3 名前・アイコン・等級・結果を混ぜない
- 名前は名前
- アイコンはアイコン
- 等級は等級
- 結果は結果

役割を分離する

### 3.4 予告と結果を分ける
- 予告は記号
- 結果は数値

この分離はDOTMON UI言語の中核ルールとする

---

## 4. 記号体系

## 4.1 選択カーソル
### 採用記号
- `▶`

### 意味
- 現在選択中の項目

### 使用箇所
- メインメニュー
- FOOD一覧
- SKILL一覧
- STAT項目
- 各種選択リスト

### 備考
- 点滅あり
- 移動時フラッシュあり
- 文字基準で位置を合わせる

---

## 4.2 増減記号
DOTMONにおける増減の**予告表現**は記号で統一する。

### 上昇
- `+` = 小上昇
- `++` = 中上昇
- `+++` = 大上昇

### 減少
- `-` = 小減少
- `--` = 中減少
- `---` = 大減少

### 使用用途
- TRN右下ペイン
- FOOD使用前の傾向表示
- ITEM効果の簡易説明
- SIGNAL / DRIVE / 補助効果の簡易説明
- 各種プレビューUI

### 例
- `SIG ++`
- `STA --`
- `STB +`
- `WGT +`

### 原則
- **予告は記号**
- 数値は出さない

---

## 4.3 数値変化表示
DOTMONにおける変化の**実結果表示**は数値で統一する。

### 形式
- `+2`
- `-1`
- `+0.5`

### 使用用途
- FOOD使用結果
- HEAL使用結果
- ログ
- STAT変化
- BTTL結果
- 実際に発生した効果の記録

### 例
- `充足値 +2`
- `安定度 +1`
- `体重 +0.5`
- `damage -1`

### 原則
- **結果は数字**
- 記号だけでは終わらせない

---

## 4.4 等級記号
等級（rank）は、名前の一部ではなく**UI上の補助情報**として扱う。

### 表示形式
- `I`
- `II`
- `III`

### 内部値
- `0` = 表示なし
- `1` = `I`
- `2` = `II`
- `3` = `III`

### 使用用途
- FOODアイコン
- ITEMアイコン
- MATERIALアイコン
- 将来のランク付きオブジェクト全般

### 表示位置
- アイコン右下寄り
- 端ぴったりではなく 1px 程度内側

### 原則
- 名前には含めない
- アイコン本体に焼き込まない
- 文字描画で後乗せする

---

## 4.5 所持数表示
### 形式
- `x00`
- `x01`
- `x99`

### 使用用途
- FOOD一覧
- ITEM一覧
- 所持物表示

### 原則
- 数量は右揃え寄り
- 名称と混ぜない
- 視線導線として「名前 → 数量」の順で読む配置を優先する

---

## 5. 略称体系

DOTMONでは小画面表示のため、内部状態や効果傾向には短い略称を使う。  
ただし、略称は**使い回し可能で一貫性のあるもの**に限る。

### 5.1 現在採用・採用候補
- `HP` = 体力
- `STA` = スタミナ
- `BRK` = BREAK
- `SIG` = 信号品質 / 信号寄り指標
- `STB` = 安定度
- `WGT` = 体重
- `DMG` = 損傷
- `FUL` = 充足値（必要時のみ）
- `QTY` = 数量（必要時のみ）

### 5.2 原則
- 3〜4文字程度に収める
- 画面ごとに別略称へしない
- 世界観用語と可読性の中間を取る

---

## 6. アイコン言語

### 6.1 基本方針
アイコンは「絵」ではなく「記号」として設計する。

### 6.2 優先順位
1. シルエットで何か分かる
2. 1bitでも読める
3. 同系統内の差が出せる
4. 必要なら等級を後乗せできる

### 6.3 同系統アイコンの扱い
同系統アイテムはベースシルエットを共有し、差分は以下で表す。

- 名前
- 説明
- 等級 (`I / II / III`)

### 6.4 例
- 肉系 → 骨付き肉シルエット
- ドリンク系 → 瓶シルエット
- ゼリー系 → 丸容器 / 塊
- 修復系 → メディア / 記録媒体シルエット

---

## 7. 画面ごとの文法

## 7.1 FOOD
### 文法
- 一覧 = 名前 + 所持数
- 右ペイン = アイコン + 等級
- 下ペイン = 名前 + 種別 + 効果 + 所持数 + 短文説明

### 予告/結果
- 一覧 / 説明 → 必要なら記号
- 実際に与えた結果 → 数字

---

## 7.2 TRN
### 文法
- 左 = モード選択
- 右 = 個体表示 + 効果傾向
- 右下 = 予告記号

### 例
- `SIG ++`
- `STA --`

### 原則
- TRN右下は「予告」なので数値ではなく記号を使う

---

## 7.3 BTTL
### 文法
- 左 = 戦闘本体
- 右 = `SIGNAL MENU` + `OBS LOG`
- 下 = 日本語ログ

### 表現
- SIGNAL / DRIVE の簡易効果説明は記号を使ってよい
- 実際の被ダメ・結果は数値やログで見せる

---

## 7.4 STAT
### 文法
- 現在値 = 数値 / バー
- 傾向 = バー / 一覧
- 詳細説明 = 下ペイン

### 原則
- STATでは「予告」より「現在状態」が主
- そのため記号よりも数値/バーを優先する

---

## 7.5 ITEM / MATERIAL / 取得報酬
### 文法
- 名前
- アイコン
- 等級
- 所持数
- 効果 / 説明

### 原則
- FOODと同じUI文法へ寄せる
- ランク付きなら右下に `I / II / III`

---

## 8. 世界観とUIの距離感

DOTMONのUIは、世界観としては
- 観測
- 管理
- 干渉
を含むが、

見た目の第一印象は
**「携帯育成液晶ゲーム風」**
を優先する。

したがって、UI文法は以下の順で優先する。

1. 一目で分かる
2. 育成玩具として読める
3. その上で世界観の冷たさや端末感を滲ませる

### つまり
- UIの表層は玩具寄り
- 語彙やログの奥側で観測端末感を出す

---

## 9. 実装上の共通責務

### 9.1 記号変換責務
- `rank -> I / II / III`
- `delta -> + / ++ / +++ / - / -- / ---`

### 9.2 描画責務
- アイコン描画
- 等級描画
- 数値表示
- 記号表示

### 9.3 分離原則
- ロジック層は意味を持つ
- UI層は記号へ変換する
- 直接画面ごとにベタ書きしすぎない

---

## 10. 禁止・非推奨

### 非推奨
- 予告と結果を同じ記法で出す
- 名前に等級表記を混ぜる
- 同じ意味なのに画面ごとに別記号を使う
- アイコン本体へ等級を焼き込む
- 略称を画面ごとに変える

### 禁止寄り
- 1つの画面で意味の違う `+` を混在させる
- 小画面で読めない装飾記号を増やす
- 記号体系をカテゴリごとのローカルルールにする

---

## 11. 今後の拡張余地

将来的には以下も同じUI言語で吸収可能。

- 状態異常記号
- レアリティ記号
- NEW表示
- 特異個体マーキング
- SIGNAL/DRIVE状態アイコン
- ADV報酬種別記号

ただし現時点では未採用。  
まずは以下を正本とする。

- 選択カーソル `▶`
- 増減記号 `+ / ++ / +++ / - / -- / ---`
- 結果数値 `+n / -n`
- 等級 `I / II / III`
- 所持数 `xNN`

---

## 12. 一言要約

DOTMON UI言語の基本文法は以下。

- **予告は記号**
- **結果は数字**
- **等級はアイコン右下**
- **名前と等級は分離**
- **同じ意味は同じ記号で表す**

この文法を、今後の全画面で共通運用する。
